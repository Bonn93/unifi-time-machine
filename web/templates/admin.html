Output Format Changes Are CPU-Intensive

Changing the Output Format below will trigger a full re-encode of every timelapse on the next scheduled video run. Depending on the number of timelapses and your hardware, this can take several minutes to hours and will significantly load the CPU. Existing files in the old format are kept until the new format is confirmed. Avoid changing format repeatedly in a short period.

Video Output

WebM/AV1 produces the smallest files but requires software decoding on most phones and older devices, which can cause buffering. MP4/H.264 uses universal hardware decoding — smooth on all devices, slightly larger files. HLS/H.264 is the best choice for mobile and remote viewing: it streams in short segments so playback starts immediately, and it adapts quality to your connection speed.
Sets the CRF (Constant Rate Factor) used by the encoder. Lower CRF = better visual quality but larger files and longer encode times. Low = CRF 35 (~small files), Medium = CRF 28 (balanced), High = CRF 20 (sharp), Ultra = CRF 15 (near-lossless). For timelapse content, Medium is a good default.
A hard ceiling on the encoded bitrate (e.g. 2M = 2 Mbps, 4M = 4 Mbps). The CRF quality setting still applies — this cap only kicks in during high-motion scenes to prevent file size spikes. Lower values mean smaller files and less bandwidth required to stream, but may cause visible quality drops during fast movement. 2M–4M is suitable for most home setups.
Controls the speed vs compression efficiency tradeoff. Fast finishes quickly using more CPU cycles per second but less total time — ideal for Raspberry Pi or NAS devices. Slow achieves smaller file sizes at the same quality level but can take 3–5× longer to encode. Unless your hardware is powerful, keep this on Fast.
How often a new snapshot is captured from the camera. 3600 = one frame per hour, giving a 24-frame daily timelapse. Lower values produce smoother, longer timelapses but consume more disk space and camera bandwidth. Changes take effect on the next scheduler tick (within one interval).

HLS Adaptive Streaming

Each HLS stream is split into short segments of this duration. Shorter segments (2–4s) start playback faster; longer segments (6–10s) reduce the number of files but delay initial playback. 4 seconds is the standard for on-demand (VOD) content.
Comma-separated list of quality levels to encode into the HLS stream. source = original camera resolution (no downscale). 1080p / 720p / 480p = downscaled variants for slower connections. The player auto-selects the best level based on available bandwidth (ABR). Encoding more levels increases CPU time and disk usage proportionally. Example: source,720p or source,1080p,720p,480p.

Timelapse Generation & Retention

Number of calendar days for which a full 24-hour daily timelapse is generated and kept. Each daily timelapse includes every snapshot from midnight to midnight. Older ones beyond this limit are deleted automatically on the next cleanup run.
Number of Monday-to-Sunday weekly timelapses to retain. Each covers one calendar week using daylight-filtered frames only. Older weeks are pruned automatically.
Number of calendar-month timelapses to retain. Each month uses one representative frame per day (closest to the target hour). Older months are pruned automatically.
Individual snapshot images (.jpg files) are kept for this many days before deletion. Longer retention allows regenerating timelapses further into the past but uses proportionally more disk space. At one snapshot per hour, 30 days ≈ 720 images.
Photos saved to the gallery are kept for this many days before automatic deletion.
How long a publicly shareable link remains valid after creation. Set to 0 for links that never expire. Expired links return a 404 and are cleaned up automatically.

Daylight Filtering

Frames captured before this hour (24h clock) are excluded from weekly, monthly, and yearly timelapses. Set to 0 to include all hours. The full 24-hour daily timelapse is always unaffected.
Frames captured after this hour are excluded from weekly, monthly, and yearly timelapses. Set to 24 (or match start hour) to disable filtering. Default: 7 to 19 covers typical daylight hours.
When building a monthly timelapse, one representative frame is chosen per day. This setting picks the snapshot closest to this hour (e.g. 12 = noon). Choosing midday avoids shadows at the extremes of the day.
Settings are saved immediately. Timelapse regeneration is queued automatically if the output format changed.
